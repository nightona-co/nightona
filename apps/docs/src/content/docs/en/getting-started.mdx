This section introduces core concepts, common workflows, and next steps for using Nightona.

## Dashboard

[Nightona Dashboard ↗](https://app.daytona.io/) is a visual user interface where you can manage sandboxes, access API keys, view usage, and more.
It serves as the primary point of control for managing your Nightona resources.

## SDKs

Nightona provides [Python](/docs/en/python-sdk), [TypeScript](/docs/en/typescript-sdk), [Ruby](/docs/en/ruby-sdk), [Go](/docs/en/go-sdk), and [Java](/docs/en/java-sdk) SDKs to programmatically interact with sandboxes. They support sandbox lifecycle management, code execution, resource access, and more.

## CLI

Nightona provides command-line access to core features for interacting with Nightona Sandboxes, including managing their lifecycle, snapshots, and more. To interact with Nightona Sandboxes from the command line, install the Nightona CLI:

<Tabs syncKey="language">
<TabItem label="Mac">

```bash
brew install nightona-co/cli/nightona
```

Trust the tap once so routine `brew upgrade` keeps the Nightona CLI up to date. Recent Homebrew versions require third-party taps to be explicitly trusted; without it, a bare `brew upgrade` skips the Nightona tap and the CLI goes stale:

```bash
brew trust nightona-co/cli
```

To upgrade the Nightona CLI to the latest version:

```bash
brew upgrade nightona-co/cli/nightona
```

Alternatively, install directly without Homebrew:

For Apple Silicon (`arm64`):

  ```bash
  sudo curl -fL https://github.com/nightona-co/nightona/releases/latest/download/nightona-darwin-arm64 -o /usr/local/bin/nightona && sudo chmod +x /usr/local/bin/nightona
  ```

For Intel (`amd64`):

  ```bash
  sudo curl -fL https://github.com/nightona-co/nightona/releases/latest/download/nightona-darwin-amd64 -o /usr/local/bin/nightona && sudo chmod +x /usr/local/bin/nightona
  ```

</TabItem>
<TabItem label="Linux">

Choose the command for your Linux architecture. Both commands download the latest binary from GitHub releases and install it to `/usr/local/bin`, overwriting any existing version.

For `amd64` (`x86_64`):

  ```bash
  sudo curl -fL https://github.com/nightona-co/nightona/releases/latest/download/nightona-linux-amd64 -o /usr/local/bin/nightona && sudo chmod +x /usr/local/bin/nightona
  ```

For `arm64` (`aarch64`):

  ```bash
  sudo curl -fL https://github.com/nightona-co/nightona/releases/latest/download/nightona-linux-arm64 -o /usr/local/bin/nightona && sudo chmod +x /usr/local/bin/nightona
  ```

</TabItem>
<TabItem label="Windows">

```bash
powershell -Command "irm https://get.daytona.io/windows | iex"
```

After installing the Nightona CLI, use the `nightona` command to interact with Nightona sandboxes from the command line.

To view all available commands and flags, see the [CLI reference](/docs/en/tools/cli).

## API

Nightona provides a RESTful API for interacting with Nightona Sandboxes, including managing their lifecycle, snapshots, and more. 
It serves as a flexible and powerful way to interact with Nightona from your own applications.

To interact with Nightona Sandboxes from the API, see the [API reference](/docs/en/tools/api).

## MCP server

Nightona provides a Model Context Protocol (MCP) server that enables AI agents to interact with Nightona Sandboxes programmatically. The MCP server integrates with popular AI agents including Claude, Cursor, and Windsurf.

To set up the MCP server with your AI agent:

```bash
nightona mcp init [claude/cursor/windsurf]
```

For more information, see the [MCP server documentation](/docs/en/mcp).

## Multiple runtime support

The [TypeScript SDK](/docs/en/typescript-sdk) ships as a dual ESM/CJS package and works out of the box in **Node.js**, **Bun**, **Next.js**, **Nuxt.js**, **Remix**, **Vite SSR**, **AWS Lambda**, and **Azure Functions** without any extra configuration.

For **Cloudflare Workers**, set the Node.js compatibility flag in your `wrangler.toml`:

```toml
compatibility_flags = ["nodejs_compat"]
```

For **Deno**, install with `deno add npm:@nightona/sdk` or import directly with the `npm:` specifier:

```typescript
import { Nightona, Image } from 'npm:@nightona/sdk'
```

For **browser apps with Vite** (or any browser bundler), install [`vite-plugin-node-polyfills`](https://www.npmjs.com/package/vite-plugin-node-polyfills) and add it to your `vite.config.ts`:

```typescript
import { defineConfig } from 'vite'
import { nodePolyfills } from 'vite-plugin-node-polyfills'

export default defineConfig({
  plugins: [nodePolyfills({ globals: { Buffer: true, process: true, global: true } })],
})
```

The SDK uses Node's `Buffer` for binary data (downloaded files, multipart bodies). Browsers don't ship `Buffer`, so the polyfill provides it. Without it, basic operations like `Image.base()` and `nightona.list()` still work, but methods that handle binary payloads (`fs.downloadFile`, `fs.downloadFiles`) will throw.

Some runtimes don't expose the full set of Node.js APIs (browsers and edge runtimes have no filesystem, no `crypto`, etc.). Methods that depend on those APIs throw a clear runtime error instead of silently producing wrong results.

```go
    "context"

}
```

```python
sandbox.delete()
```

</TabItem>

```typescript

await sandbox.delete();
```

</TabItem>
<TabItem label="Go" icon="seti:go">

```go
package main

import (
    "context"
    "fmt"
    "log"

    "github.com/nightona-co/nightona/libs/sdk-go/pkg/nightona"
)

func main() {
    client, err := nightona.NewClient()
    if err != nil {
        log.Fatal(err)
    }

    sandbox, err := client.Create(context.Background(), nil)
    if err != nil {
        log.Fatal(err)
    }

    response, err := sandbox.Process.ExecuteCommand(context.Background(), "echo 'Hello, World!'")
    if err != nil {
        log.Fatal(err)
    }
    fmt.Println(response.Result)
    sandbox.Delete(context.Background())
}
```

### Create a sandbox with volumes

Create a sandbox with a [volume](/docs/en/volumes) mounted to share data across sandboxes.

<Tabs syncKey="language">
<TabItem label="Python" icon="seti:python">

```python
from nightona import Nightona, CreateSandboxFromSnapshotParams, VolumeMount

nightona = Nightona()
volume = nightona.volume.get("my-volume", create=True)
sandbox = nightona.create(
    CreateSandboxFromSnapshotParams(
        volumes=[VolumeMount(volume_id=volume.id, mount_path="/home/daytona/data")]
    )
)
```

</TabItem>
<TabItem label="TypeScript" icon="seti:typescript">

```typescript
import { Nightona } from '@nightona/sdk';

const nightona = new Nightona();
const volume = await nightona.volume.get('my-volume', true);
const sandbox = await nightona.create({
    volumes: [{ volumeId: volume.id, mountPath: '/home/daytona/data' }]
});
```

</TabItem>
<TabItem label="Go" icon="seti:go">

```go
package main

import (
    "context"
    "log"

    "github.com/nightona-co/nightona/libs/sdk-go/pkg/nightona"
    "github.com/nightona-co/nightona/libs/sdk-go/pkg/types"
)

func main() {
    client, err := nightona.NewClient()
    if err != nil {
        log.Fatal(err)
    }

    volume, err := client.Volume.Get(context.Background(), "my-volume")
    if err != nil {
        volume, err = client.Volume.Create(context.Background(), "my-volume")
        if err != nil {

            }},
        },
```
