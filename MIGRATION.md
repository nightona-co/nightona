# Migrating from Daytona v0.190.0 to Nightona

This guide is for operators of a **self-hosted Daytona v0.190.0 deployment** (Docker Compose or Kubernetes) who want to move to Nightona, and for developers whose applications use the Daytona SDKs or CLI against that deployment.

Nightona starts from the exact Daytona v0.190.0 codebase, so the runtime behavior, API surface, and data model are unchanged. What changes is **naming**: binaries, packages, module paths, environment variables, and configuration locations. This document lists every rename and the compatibility shims that ease the transition.

> **Bootstrap-phase disclaimer**: Nightona has not yet published its own packages or container images (see [PUBLISHING.md](PUBLISHING.md)) — build from this repository. The compatibility behavior described below is implemented in this tree and each claim is verified against the source files cited. End-to-end migration of real production data has **not** been exercised yet; treat the [Persisted state](#persisted-state) section as a checklist, not a guarantee. Remaining cosmetic rebrand items are tracked in [BRANDING-TODO.md](BRANDING-TODO.md).

## TL;DR checklist

1. Back up your PostgreSQL database, Redis data, and object storage before switching.
2. Point your deployment at Nightona builds (built from this repository — no prebuilt Nightona artifacts exist yet).
3. Keep your existing `DAYTONA_*` environment variables — they continue to work — and migrate to `NIGHTONA_*` at your own pace.
4. Update SDK dependencies to the Nightona package names when they are published; until then, build the SDKs from source.
5. Set the API URL explicitly in SDK configuration or via `NIGHTONA_API_URL` — the default is no longer the Daytona SaaS.
6. Review the [Persisted state](#persisted-state) section before starting the new stack against old data — in particular the Docker Compose project rename, which changes volume names and the Postgres database name.

## Environment variables

All `DAYTONA_*` environment variables have `NIGHTONA_*` equivalents. A compatibility shim keeps the legacy names working:

- **`NIGHTONA_*` is preferred.** New documentation, examples, and defaults use the `NIGHTONA_` prefix.
- **`DAYTONA_*` is still honored.** When a component looks up `NIGHTONA_FOO` and it is unset, it falls back to the legacy `DAYTONA_FOO` twin. If both are set, `NIGHTONA_*` wins.
- The TypeScript, Python, and Ruby SDKs also apply the fallback to values read from `.env` / `.env.local` files (precedence: runtime env, then `.env.local`, then `.env`; the `NIGHTONA_*` name wins over its `DAYTONA_*` twin).

Common variables:

| Legacy (Daytona) | Preferred (Nightona) |
| ---------------------------- | ------------------------------ |
| `DAYTONA_API_KEY` | `NIGHTONA_API_KEY` |
| `DAYTONA_API_URL` | `NIGHTONA_API_URL` |
| `DAYTONA_SERVER_URL` (deprecated alias) | `NIGHTONA_SERVER_URL` (deprecated alias) |
| `DAYTONA_JWT_TOKEN` | `NIGHTONA_JWT_TOKEN` |
| `DAYTONA_ORGANIZATION_ID` | `NIGHTONA_ORGANIZATION_ID` |
| `DAYTONA_TARGET` | `NIGHTONA_TARGET` |
| `DAYTONA_CONFIG_DIR` | `NIGHTONA_CONFIG_DIR` |

### Where the fallback is implemented

- **Go binaries** — `nightona` CLI, `daemon`, `proxy`, `runner`, `snapshot-manager`, `ssh-gateway`: each calls an alias shim as the first thing in `main()` that copies every `DAYTONA_<X>` to `NIGHTONA_<X>` (unless already set) and prints a one-time deprecation warning. The shared implementation is `libs/common-go/pkg/envcompat`; the CLI, ssh-gateway, and snapshot-manager carry an inlined copy (`apps/cli/envcompat.go`, `apps/ssh-gateway/envcompat.go`, `apps/snapshot-manager/cmd/envcompat.go`) because their builds do not depend on `libs/common-go`.
- **OpenTelemetry collector** — the collector binary is generated by the OTel Collector Builder, so the shim runs from an `init()` in `apps/otel-collector/exporter/envcompat.go`, before the collector resolves `${env:...}` references in its config.
- **API server (NestJS)** — `apps/api/src/legacy-env.ts`, kept as the very first import in `apps/api/src/main.ts` so it runs before tracing/config code reads the environment.
- **Go SDK** — `envOrLegacy` in `libs/sdk-go/pkg/nightona/envcompat.go`; every `NIGHTONA_*` lookup in the client falls back to its `DAYTONA_*` twin.
- **TypeScript SDK** — `NightonaEnvReader` in `libs/sdk-typescript/src/utils/Runtime.ts` (covers runtime env plus `.env` / `.env.local` on Node).
- **Python SDK** — `NightonaEnvReader` in `libs/sdk-python/src/nightona/_utils/env.py` (used by both the sync and async clients; covers runtime env plus `.env` / `.env.local`).
- **Ruby SDK** — `Nightona::Config` in `libs/sdk-ruby/lib/nightona/config.rb` (covers runtime env plus `.env` / `.env.local`).
- **Java SDK** — `io.nightona.sdk.Nightona`'s default-config factory reads `NIGHTONA_API_KEY`, `NIGHTONA_API_URL`, and `NIGHTONA_TARGET` with `DAYTONA_*` fallbacks (`envWithFallback` in `Nightona.java`).

**Known limitation — `.env` files read by Go binaries**: the CLI (`godotenv.Load()` in `apps/cli/main.go`) and the proxy (`godotenv.Overload(...)` in `apps/proxy/cmd/proxy/config/config.go`) load `.env` files *after* the alias shim has already run. A `DAYTONA_*` key that exists **only in a `.env` file** (not in the process environment) is therefore **not** mirrored to `NIGHTONA_*` for those two binaries. Use the `NIGHTONA_*` name in `.env` files, or export the variable in the environment. Setting both names is safe in all components.

## CLI configuration directory

The CLI (now installed as `nightona` instead of `daytona`) stores its configuration under a `nightona` directory inside your platform's user config directory (e.g. `~/.config/nightona` on Linux, `~/Library/Application Support/nightona` on macOS), or wherever `NIGHTONA_CONFIG_DIR` points (`DAYTONA_CONFIG_DIR` still works via the env-var shim above).

**Legacy fallback (implemented)**: if the `nightona` config directory does not exist yet but the legacy `daytona` one does (same parent directory, e.g. `~/.config/daytona`), the CLI keeps using the legacy directory and prints a one-time deprecation warning to stderr telling you to move it (`GetConfigDir()` in `apps/cli/config/config.go`). Existing profiles and API keys therefore keep working with no action on your part. No automatic copy or migration is performed.

To silence the warning and complete the move, migrate manually at any time — the file format is unchanged:

```bash
# Linux example; adjust paths for your platform
mkdir -p ~/.config/nightona
cp -r ~/.config/daytona/* ~/.config/nightona/
```

## Package renames

Nightona-named packages are **not published yet** (see [PUBLISHING.md](PUBLISHING.md)); until they are, build from this repository. The planned coordinates:

| Ecosystem | Daytona v0.190.0 (upstream, published) | Nightona (this repo / planned) |
| --------- | -------------------------------------- | ------------------------------ |
| npm (TypeScript SDK) | `@daytonaio/sdk` | `@nightona/sdk` |
| PyPI (Python SDK) | `daytona` | `nightona` |
| RubyGems (Ruby SDK) | `daytona` | `nightona` |
| Go module (Go SDK) | `github.com/daytonaio/daytona/libs/sdk-go` | `github.com/Amartuvshins0404/nightona/libs/sdk-go` |
| Java / Maven (Java SDK) | `io.daytona:sdk` | `io.nightona:sdk` (with `io.nightona:api-client`, `io.nightona:toolbox-api-client`) |
| CLI binary | `daytona` | `nightona` |

Import paths and top-level namespaces change accordingly, e.g.:

```python
# before
from daytona import Daytona
# after
from nightona import Nightona
```

```typescript
// before
import { Daytona } from '@daytonaio/sdk'
// after
import { Nightona } from '@nightona/sdk'
```

## SDK default API URL

The Daytona SDKs defaulted to the Daytona SaaS endpoint (`https://app.daytona.io/api`). Nightona has no SaaS, so the SDK default is now the local self-hosted API:

```
http://localhost:3000/api
```

This default is in place in **all five SDKs**:

| SDK | Where the default lives |
| --- | ----------------------- |
| Go | `defaultAPIURL` in `libs/sdk-go/pkg/nightona/client.go` |
| TypeScript | constructor fallback in `libs/sdk-typescript/src/Nightona.ts` |
| Python (sync and async) | `default_api_url` in `libs/sdk-python/src/nightona/_sync/nightona.py` and `_async/nightona.py` |
| Ruby | `Nightona::Config::API_URL` in `libs/sdk-ruby/lib/nightona/config.rb` |
| Java | `NightonaConfig` builder default and the env factory in `libs/sdk-java/src/main/java/io/nightona/sdk/` |

No SDK in this tree defaults to `https://app.daytona.io/api` anymore. If your API is reachable elsewhere, set it explicitly — either `NIGHTONA_API_URL` (or legacy `DAYTONA_API_URL`) in the environment, or `apiUrl` / `api_url` in the SDK constructor configuration.

## Container images

Nightona does not yet operate its own image registries. Deployments continue to pull the **upstream published Daytona images** where images are referenced at runtime, most notably the default sandbox snapshot image:

- `daytonaio/sandbox:<tag>` (e.g. `DEFAULT_SNAPSHOT=daytonaio/sandbox:0.5.0-slim` in `docker/docker-compose.yaml`)
- `ghcr.io/daytonaio/...` images referenced by charts/compose files

These are unmodified upstream artifacts pulled from Daytona's registries and will keep working as long as upstream keeps them published. Once Nightona publishes its own images (see the [roadmap](README.md#roadmap)), the references will be switched and this guide updated. If you need to insulate yourself from upstream registry availability now, mirror the images into your own registry and override the references.

## Persisted state

### Docker Compose: project rename changes volume names and the database name

`docker/docker-compose.yaml` now sets the Compose project name to `nightona` (top-level `name:`; upstream used `daytona`). Compose namespaces volumes by project name, so a plain `docker compose up` on the new file creates and uses **new, empty volumes** — `nightona_db_data`, `nightona_minio_data`, `nightona_registry`, `nightona_dex_db` — and does **not** touch your existing `daytona_db_data`, `daytona_minio_data`, `daytona_registry`, `daytona_dex_db` volumes. The old volumes are left orphaned on disk (your data is still there, but the new stack starts empty).

Additionally, the Postgres database name changed: the compose file sets `POSTGRES_DB=nightona` (db service) and `DB_DATABASE=nightona` (api service); upstream used `daytona`. Note that `POSTGRES_DB` only takes effect when Postgres initializes an **empty** data volume — a volume carried over from the old stack still contains a database named `daytona`, which the API will not find if `DB_DATABASE=nightona`.

Pick one of these migration paths:

**Option A — keep the old volumes by overriding the project name.** `-p` beats the file-level `name:`:

```bash
docker compose -p daytona up -d
```

This reuses the existing `daytona_*` volumes as-is. You must also override the api service's database name back to the existing one, e.g. via a compose override file or `DB_DATABASE=daytona` in the api environment (the db service's `POSTGRES_DB` is ignored on an already-initialized volume).

**Option B — copy the volumes to the new names** (with the old stack stopped):

```bash
for v in db_data minio_data registry dex_db; do
  docker volume create "nightona_${v}"
  docker run --rm \
    -v "daytona_${v}":/from:ro \
    -v "nightona_${v}":/to \
    alpine sh -c 'cd /from && cp -a . /to'
done
```

Then rename the database inside the copied volume to match `DB_DATABASE=nightona`:

```bash
docker compose up -d db
docker compose exec db psql -U user -d postgres \
  -c 'ALTER DATABASE "daytona" RENAME TO "nightona";'
docker compose up -d
```

(Alternatively, keep the old database name and set `DB_DATABASE=daytona` on the api service.) Keep the `daytona_*` volumes around until the new stack has run cleanly; remove them later with `docker volume rm`.

### Renamed identifiers in application state

The case-preserving rename was applied across the whole source tree, which means some **identifier strings that end up in persisted state** differ between a Daytona v0.190.0 deployment's data and what the renamed Nightona code expects. Confirmed in this tree:

- **Seeded database identifiers**: the admin user id is `nightona-admin` (`NIGHTONA_ADMIN_USER_ID` in `apps/api/src/app.service.ts`), and migration `1742475055353` updates the row `WHERE id = 'nightona-admin'`; an existing Daytona database contains `daytona-admin`.
- **Kafka consumer group**: the API consumer `groupId` is `nightona` (`apps/api/src/main.ts`); upstream used `daytona`. A group id change resets consumer offsets for that group.

Not yet audited exhaustively:

- **Queue / Redis / job identifiers**: queue names, Redis key prefixes, lock names, and scheduled-job identifiers that embed the product name may have changed the same way, orphaning in-flight jobs or cached state from the old deployment.
- **TypeORM migration history**: the `migrations` table records executed migrations by name/timestamp; these are expected to be unchanged, but verify that the renamed codebase does not attempt to re-run or rename any migration.

**Before migrating production data**, diff the identifier strings your deployment persists (database rows, Redis keys, Kafka topics/groups, object-storage paths) against what the Nightona build emits, and prepare data-fix SQL / key-rename steps where they diverge. An end-to-end migration on real data has not been validated yet. Recommended approach:

1. Take full backups (PostgreSQL dump, Redis RDB/AOF, object storage).
2. Restore into a staging environment and start the Nightona stack against it.
3. Verify: admin login, existing organizations/users, sandbox listing, snapshot operations, and background job processing.
4. Only then cut over production, keeping the backups until the new stack has run cleanly.
